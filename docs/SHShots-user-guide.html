01

What this plugin does

2 min read

You write KB articles. Articles need screenshots. Getting a clean BackOffice screenshot used to mean paid tools, sandbox login walls, or cropping by hand. SHShots removes all of that — you ask Claude in plain English, and a finished, annotated, KB-ready PNG comes back.

🔓

Pre-GA screens, no login wall

It drives your own Chrome window — the one you're already logged into. Feature-flagged and pre-release pages you can see, it can capture. Your password is never typed, stored, or seen by the AI.

🎯

Finds the right window

Targets the Chrome window by its URL — not just "whatever's in front". Your other windows and tabs are left alone, and the screenshot never grabs your bookmarks bar.

⏱️

Spinner-proof timing

It waits for the page to confirm it has fully finished loading before shooting — so no half-loaded charts or skeleton screens in your shot.

✅

Self-verifying

After every capture, Claude looks at the image and confirms it actually shows what you asked for — wrong window, black screen, or wrong page get caught and re-shot, not sent to you broken.

🖍️

Annotates in the house style

Red rounded boxes, tapered arrows, blur for sensitive data, rounded corners — the exact style used across StoreHub KB articles (more in chapter 06).

📜

Full-page capture

Pages taller than the screen are captured in slices and stitched into one seamless full-page image — scrollbars and floating buttons cleaned out of the seam.

What it will NOT do

  • It never logs in for you. You log into BackOffice yourself. The plugin only drives a window that is already logged in.
  • It never stores credentials. There is no password field, no token, nothing saved.
  • It won't capture while you're mid-click. Captures need the Chrome window in front for ~20–30 seconds — Claude will tell you before it starts (see chapter 05).

Install

~1 minute

Two commands, typed inside Claude Code (not the plain terminal). The first is a one-time pointer that tells Claude Code where the plugin lives — the repo address is baked into the command, so there's no GitHub link to visit, nothing to download, and no GitHub account needed. The second installs (and later updates) the plugin.

1

Point Claude Code at the repo (one time only)

> /plugin marketplace add AndreaAK8/SHShots
2

Install the plugin

> /plugin install backoffice-screenshot@shshots
# later, to update to the newest version, run the same command again
✅
That's it for install. Restart your Claude Code session and the skill is live. Next: a few one-time macOS permissions (chapter 04) — Claude walks you through these the first time you ask for a screenshot, so you can also just skip ahead and try it.
04

First-run setup

one-time · ~3 minutes

macOS protects your screen and your browser from automation — rightly so. The first capture needs three approvals. You only ever do this once per machine.

1

Automation → Google Chrome

The first time the plugin talks to Chrome, macOS shows a popup: "…wants to control Google Chrome". Click Allow. If you accidentally clicked Don't Allow, fix it under System Settings → Privacy & Security → Automation → enable Google Chrome for your terminal app.

2

Screen Recording for your terminal

Go to System Settings → Privacy & Security → Screen & System Audio Recording and switch ON the app you run Claude Code in (Terminal, iTerm, or Visual Studio Code). Then quit and reopen that app — the grant only takes effect after a restart.

⬛
Skipped this? Every screenshot comes out pure black. It's the #1 first-run issue — see Troubleshooting.
3

Let Chrome accept JavaScript from automation

The plugin reads the page (Is it loaded? Where is that button?) by asking Chrome to run tiny snippets of JavaScript. Chrome ships with this switched off. Turn it on either from Chrome's menu bar: View → Developer → Allow JavaScript from Apple Events, or — if that menu doesn't stick — run this in your terminal and restart Chrome:

$ defaults write com.google.Chrome AllowJavascriptAppleEvents -bool true
🔍
This setting only allows page-reading and scrolling by tools you run on your own Mac. It does not expose your browser to websites or anything remote.
05

Daily use

the fun part

Open BackOffice in Chrome, log in, then just tell Claude what you want. Plain English. It handles navigation, waiting, capture, verification, and annotation.

Example asks

> Capture the Promotions list page in BackOffice

> Screenshot the Wastage Insights dashboard — full page, and blur anything sensitive

> Capture the Employees page. Box the 'Employees' item in the sidebar and the '+ Add Employee' button, with arrows.

> I have a KB draft with [SCREENSHOT: ...] placeholders — capture all of them from this BackOffice page and follow the instructions in each placeholder.

How a capture session flows

1

You log in

Open the BackOffice page (any environment — production, test, sandbox) in Chrome and log in yourself. Tell Claude which page or paste the URL.

2

Claude checks the page

It finds your window by URL, confirms the page has fully loaded, and locates the exact elements it needs to frame or annotate. If a section name is ambiguous, it asks you first instead of guessing.

3

Hands off for ~20–30 seconds

Claude tells you it's about to capture and brings the Chrome window to the front. Don't switch apps or click during this window — the capture photographs your actual screen, so whatever's in front is what gets shot. When it says done, the screen is yours again.

4

Verify → annotate → deliver

Claude looks at each capture to confirm it's correct, applies the annotations you asked for, rounds the corners, blurs anything sensitive, and hands you the finished PNG — or inserts it directly into your Lark doc if you asked for that.

💡
Pro move: batch your asks. "Capture these 7 sections" runs as one ~25-second screen session instead of seven interruptions.

Asking for annotations

Describe what to point at the way you'd tell a colleague — Claude finds the pixel coordinates itself:

  • "Box the 'Apply' button and point an arrow at it"
  • "Arrow pointing at the 'Daily' dropdown on the right of the chart"
  • "Blur the email and phone columns — they show real staff data"
  • "Crop to just the filter row at the top"

The annotation style

locked · do not freestyle

Every mark the plugin draws follows one locked house style, so screenshots look identical no matter who captured them. You never pick colors or shapes — consistency is the whole point.

SH RED
#EA3329 — the only annotation color
BOX
Rounded-corner rectangle, 4px stroke
⬅
Tapered arrow — thin tail, solid head
▒▒ BLUR ▒▒
Gaussian blur over sensitive regions
ElementSpec
Color#EA3329 red — every box, every arrow. No other colors.
BoxesRounded corners (8px radius), 4px stroke, drawn around the exact UI element with a small breathing margin.
ArrowsTapered: a thin tail that widens into a solid triangular head. Points left or right, tip lands just beside the target. All edges antialiased — no jaggies.
BlurOptional Gaussian smudge over emails, phone numbers, real customer names — anything that shouldn't ship in a public article.
FinishThe screenshot itself gets gently rounded corners (transparent), matching the KB look.
Browser chromeTab bar, omnibox, and bookmarks are always stripped — the image starts at the page content.

For the curious: the CLI underneath

You normally never touch this — Claude drives it for you. But the annotation engine is a plain Python script you can also run by hand:

$ python3 scripts/annotate.py shot.png \
    --box 50,40,120,60 \ # rounded red box at X,Y sized WxH
    --arrow 200,150,right \ # tapered arrow pointing at that pixel
    --blur 1340,380,720,800 \ # smudge a sensitive region
    --crop-chrome 1440 \ # strip the browser bar (window width in points)
    --round 24 --out final.png # rounded corners + output path

Troubleshooting

when shots go wrong
SymptomCauseFix
Screenshot is pure black Screen Recording permission missing — or macOS silently revoked it (it re-asks periodically). Open System Settings → Privacy & Security → Screen & System Audio Recording, make sure your terminal app is ON. Just visiting this panel often refreshes a revoked grant. If it was OFF, toggle it on and restart the terminal app.
"Executing JavaScript through AppleScript is turned off" Chrome's automation-JS setting is off (or didn't stick from the menu). Run defaults write com.google.Chrome AllowJavascriptAppleEvents -bool true then fully quit and reopen Chrome.
Captured the wrong app / your Slack / Lark You switched apps during the hands-off window. Nothing broke — Claude's verification catches it and re-shoots. Keep hands off until it says done.
Captured the wrong Chrome window Multiple BackOffice windows open on similar pages. The plugin targets by URL, so this is rare — but tell Claude which exact URL if you have several environments open at once.
Page elements look half-loaded Very slow connection beat the load check. Ask Claude to re-capture — it re-verifies readiness each time. Mention "wait for the charts" for data-heavy dashboards.
"…wants to control Google Chrome" popup again macOS re-confirms Automation grants occasionally, or you updated your terminal app. Click Allow. One click, you're back.
Annotation points at slightly the wrong spot Your Chrome zoom level changed mid-session (coordinates are zoom-dependent). Tell Claude — it re-reads the layout and redraws. Avoid changing zoom between ask and capture.
🆘
Still stuck? Ping Andrea (AK) with the broken PNG and what you asked for — most issues are a one-line fix.
08

FAQ

quick answers
Where can I use the screenshots — only KB articles?

Anywhere. The output is a normal PNG saved on your Mac. Paste it into Jira tickets, Lark docs, Confluence pages, training decks, chat — anything that takes an image. KB articles are just the most common use. (Automatic insertion into Lark docs at [SCREENSHOT: …] placeholders is also available — see below.)

Does the AI see or store my BackOffice password?

No — by design. You log in yourself, in your own Chrome. The plugin only interacts with a window that's already authenticated. There is no flow anywhere in the plugin that asks for, types, or saves a credential.

Can it capture test / staging / pre-GA environments?

Yes — that's its superpower. If you can see the page in Chrome (feature flags, staging URLs, pre-release builds), the plugin can capture it. No sandbox login wall, because there's no separate login at all.

Why do I have to keep my hands off during capture?

The capture photographs your actual screen (that's what lets it bypass login walls). If you bring another app in front mid-shot, that app is what gets photographed. Claude verifies every frame and re-shoots if you photobomb it — but staying hands-off for ~25 seconds is faster.

Can it put screenshots straight into my Lark doc?

Yes — if your KB draft uses [SCREENSHOT: description] placeholders, ask Claude to capture them all and insert each image under its placeholder. (Requires lark-cli set up on your machine; ask AK.)

Can I change the annotation color or arrow shape?

No — the style is locked so every StoreHub KB screenshot looks the same regardless of author. If the style itself needs to evolve, raise it with AK; it changes in one place for everyone.

Does it work on Windows or with Safari?

Not currently. The pipeline is built on macOS built-ins and Chrome's AppleScript support. macOS + Chrome only.
